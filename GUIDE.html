// 01 — getting started

Stack Startup

Everything you need to bring the stack up, verify it's healthy, and know where each service lives.

First-time setup

bash
# Clone your stack, then:
cp .env.example .env
nano .env                    # Set all CHANGE_ME passwords

# Start everything (detached)
make up
# or: docker compose up -d

# Watch services come alive
docker compose logs -f       # Ctrl+C to stop watching

# Verify all health checks pass (~2 min for GeoServer)
make health

Daily operations

bash — make targets
make up            # Start stack
make down          # Stop stack (data persists in volumes)
make restart       # Restart all services
make ps            # Show container status + ports
make health        # Full health check + service URLs
make logs          # Tail all logs
make logs-db       # PostGIS logs only
make logs-gs       # GeoServer logs only
make shell-db      # psql shell into PostGIS
make backup        # Dump DB to ./backups/
make vacuum        # VACUUM ANALYZE database
⚠ GeoServer takes 60–90 seconds to fully start. make health will show it as "starting" until it's ready. This is normal — its health check has a 90s start_period.

// 03 — architecture

Data Flow Architecture

Understanding which service to use for which task.

🗄️
PostGIS
The source of truth. All vector and raster data lives here. GeoServer and Martin both read from PostGIS. Never write directly from clients — always go through a service.
🗺️
GeoServer
OGC compliance layer. Use for WMS image maps, WFS feature downloads, WCS raster access, QGIS/ArcGIS integration, and complex SLD cartography. Caches tiles via GeoWebCache.
⚡
Martin
Speed-first vector tiles. Use for MapLibre GL / Mapbox GL web apps. Zero config — auto-discovers all PostGIS tables with geometry. Tiles are rendered client-side for full interactivity.
🔀
Nginx
Routes /geoserver/* → GeoServer:8080 and /tiles/* → Martin:3000. Adds tile caching headers and CORS. Single port 80 entry point for all services.

Decision guide: GeoServer vs Martin

Use caseUse
QGIS / ArcGIS client connectionGeoServer WMS/WFS
MapLibre GL / Mapbox web mapMartin MVT
Serving a GeoTIFF as a map layerGeoServer WMS
Download features as GeoJSON/ShapefileGeoServer WFS
Interactive hover/popup on web mapMartin MVT
Complex thematic cartography (SLD rules)GeoServer WMS
Pre-seeded tile cache for high trafficGeoServer WMTS
Real-time data from PostGIS, no cacheMartin MVT
DEM / raster elevation accessGeoServer WCS
Serve PMTiles / MBTiles static filesMartin file source

// PostGIS — 05

PostGIS importing data

Load any vector or raster format into PostGIS. ogr2ogr handles virtually every vector format; raster2pgsql handles rasters.

ogr2ogr — any vector format (GeoPackage, Shapefile, GeoJSON…)

bash — recommended pattern
ogr2ogr \
  -f "PostgreSQL" \
  "PG:host=localhost port=5432 dbname=gisdb user=gisuser password=YOUR_PW" \
  input_data.gpkg \                # .shp .geojson .kml .gpkg .fgb etc.
  -nln my_layer_name \              # Output table name
  -nlt PROMOTE_TO_MULTI \            # Ensures consistent geometry type
  -lco SCHEMA=gis \                  # Target schema
  -lco GEOMETRY_NAME=geom \          # Standard column name
  -lco FID=id \                       # Primary key name
  -t_srs EPSG:4326 \                  # Reproject to WGS84 on import
  --config PG_USE_COPY YES \          # 10x faster for large datasets
  -overwrite \
  -progress

Common ogr2ogr recipes

bash — specific layer from GeoPackage
# List layers in a GeoPackage first
ogrinfo -al -so data.gpkg

# Import specific layer
ogr2ogr -f "PostgreSQL" "PG:..." \
  data.gpkg roads \            # layer name after file
  -nln roads \
  -lco SCHEMA=gis
bash — filter on import
# Import only features where type = 'highway'
ogr2ogr -f "PostgreSQL" "PG:..." \
  roads.shp \
  -where "type = 'highway'" \
  -nln highways \
  -lco SCHEMA=gis

# Clip to a bounding box on import
ogr2ogr -f "PostgreSQL" "PG:..." \
  -spat 36.5 -1.5 37.5 -0.5 \   # xmin ymin xmax ymax
  big_dataset.gpkg

raster2pgsql — DEM, GeoTIFF, satellite imagery

bash — raster import pipeline
# Basic import — tiled 256x256, with overview + constraints
raster2pgsql \
  -s 4326 \          # Source SRID
  -I \              # Create spatial index
  -C \              # Apply raster constraints (metadata)
  -M \              # Vacuum analyze after import
  -F \              # Add filename column
  -t 256x256 \      # Tile size (important for performance)
  dem.tif \
  raster.elevation \
  | psql -h localhost -U gisuser -d gisdb

# Large raster: use append mode to load tiles in batches
raster2pgsql -s 4326 -I -C -t 256x256 -a \  # -a = append
  big_raster.tif raster.satellite | psql -h localhost -U gisuser -d gisdb

# Via make shortcut:
make import-raster FILE=dem.tif TABLE=elevation_model
ℹ Always import to the staging schema first, inspect with make shell-db, then CREATE TABLE gis.my_layer AS SELECT * FROM staging.my_layer; when satisfied. This prevents dirty data from hitting your live services.

PostGIS spatial queries

The essential spatial SQL you'll use daily. All run in make shell-db or pgAdmin.

Geometry basics

sql
-- Check what's in your database
SELECT f_table_schema, f_table_name, f_geometry_column, srid, type
FROM geometry_columns
WHERE f_table_schema = 'gis';

-- Count features
SELECT COUNT(*) FROM gis.my_layer;

-- Bounding box of a layer
SELECT ST_Extent(geom) FROM gis.my_layer;

-- Reproject to Web Mercator (EPSG:3857)
SELECT id, ST_Transform(geom, 3857) AS geom_web
FROM gis.my_layer;

-- Export a feature as GeoJSON (for debugging)
SELECT ST_AsGeoJSON(geom) FROM gis.my_layer WHERE id = 1;

Spatial relationships

sql — common predicates
-- Points within a polygon
SELECT p.*
FROM gis.points p, gis.regions r
WHERE r.name = 'Nairobi'
  AND ST_Within(p.geom, r.geom);

-- Buffer + intersect (500m buffer around roads)
SELECT b.* FROM gis.buildings b
JOIN gis.roads r ON ST_Intersects(
  b.geom,
  ST_Buffer(ST_Transform(r.geom, 32737), 500)  -- buffer in metres (UTM)
);

-- Distance between points (in metres — use UTM projection)
SELECT
  a.name,
  b.name,
  ST_Distance(
    ST_Transform(a.geom, 32737),
    ST_Transform(b.geom, 32737)
  ) AS distance_m
FROM gis.facilities a, gis.facilities b
WHERE a.id != b.id
ORDER BY distance_m
LIMIT 10;

-- Nearest neighbor (KNN — very fast with GIST index)
SELECT id, name, geom <>> 'SRID=4326;POINT(36.8 -1.3)'::geometry AS dist
FROM gis.facilities
ORDER BY geom <> 'SRID=4326;POINT(36.8 -1.3)'::geometry
LIMIT 5;

Spatial aggregation & analysis

sql
-- Union all polygons by category
SELECT category, ST_Union(geom) AS geom
FROM gis.parcels
GROUP BY category;

-- Clip layer to a region boundary
SELECT ST_Intersection(a.geom, b.geom) AS geom
FROM gis.landuse a, gis.admin_boundary b
WHERE b.name = 'Nairobi County'
  AND ST_Intersects(a.geom, b.geom);

-- Create a materialized view (auto-exposed by Martin!)
CREATE MATERIALIZED VIEW gis.roads_simplified AS
SELECT
  id, name, road_type,
  ST_SimplifyPreserveTopology(geom, 0.0001) AS geom
FROM gis.roads
WHERE road_type IN ('primary', 'secondary');

-- Refresh when source data changes:
REFRESH MATERIALIZED VIEW gis.roads_simplified;

PostGIS raster operations

sql — raster queries
-- Inspect raster metadata
SELECT
  rid,
  ST_BandNoDataValue(rast) AS nodata,
  ST_MetaData(rast).*,
  ST_BandMetaData(rast, 1).*
FROM raster.elevation
LIMIT 1;

-- Extract pixel value at a point (elevation at coordinate)
SELECT ST_Value(
  rast,
  ST_SetSRID(ST_MakePoint(36.82, -1.28), 4326)
) AS elevation_m
FROM raster.elevation
WHERE ST_Intersects(
  rast,
  ST_SetSRID(ST_MakePoint(36.82, -1.28), 4326)
);

-- Zonal statistics (mean elevation per polygon)
SELECT
  p.name,
  (ST_SummaryStatsAgg(r.rast, 1, true)).mean AS mean_elevation,
  (ST_SummaryStatsAgg(r.rast, 1, true)).min  AS min_elevation,
  (ST_SummaryStatsAgg(r.rast, 1, true)).max  AS max_elevation
FROM gis.admin_regions p
JOIN raster.elevation r
  ON ST_Intersects(r.rast, p.geom)
GROUP BY p.name;

-- Export a raster tile as PNG (useful for debugging)
SELECT encode(ST_AsPNG(
  ST_Union(ST_Clip(rast, ST_MakeEnvelope(36.7,-1.4,37.0,-1.1,4326)))
), 'base64')
FROM raster.elevation;

PostGIS indexes & tuning

sql — essential indexes
-- Create GIST spatial index (always do this after import)
CREATE INDEX idx_my_layer_geom
  ON gis.my_layer USING GIST (geom);

-- Helper function (defined in init script):
SELECT create_spatial_index('my_layer', 'geom');

-- Compound index: spatial + attribute (e.g. spatial filter + type filter)
CREATE INDEX idx_roads_type_geom
  ON gis.roads USING GIST (geom, road_type)
  WHERE road_type = 'primary';

-- Check index usage
EXPLAIN ANALYZE
SELECT * FROM gis.my_layer
WHERE ST_Within(geom, ST_MakeEnvelope(36.7,-1.4,37.0,-1.1,4326));

-- CLUSTER table by spatial index (puts nearby features together on disk)
CLUSTER gis.my_layer USING idx_my_layer_geom;

-- Update statistics after bulk load
ANALYZE gis.my_layer;
VACUUM ANALYZE gis.my_layer;  -- full maintenance
// GeoServer — 09

GeoServer first setup

Configure GeoServer for the first time via its web UI.

  1. Open http://localhost:8080/geoserver/web/ — login with your GEOSERVER_ADMIN_USER / GEOSERVER_ADMIN_PASSWORD from .env
  2. Security → Change password if you haven't already. Critical for anything non-local.
  3. Server → Contact Information — fill in your organisation name. This appears in GetCapabilities responses.
  4. Data → Workspaces → Add new workspace — create a workspace (e.g. gis). This becomes the URL namespace for your services.
  5. Data → Stores → Add new store → PostGIS — connect to your database (host: postgis, db: gisdb, user: gisuser)
  6. Data → Layers → Add new layer — pick your store, then publish any layer from the gis schema
⚠ Use postgis (the container name) as the host, NOT localhost. GeoServer runs inside Docker and resolves container names via the internal gis_net network.

GeoServer WMTS & tile caching

GeoWebCache is bundled with GeoServer. Use it to pre-seed tile caches for high-traffic layers.

  1. Publish your layer normally via WMS
  2. Tile Layers → your_layer → Seed/Truncate
  3. Set zoom levels (e.g. 0–14), thread count (4), gridset (EPSG:4326 or WebMercatorQuad)
  4. Click Submit — seeding runs in background. Check Tile Layers → Disk quota for progress
url — WMTS tile (after seeding)
http://localhost:8080/geoserver/gwc/service/wmts?
  SERVICE=WMTS&VERSION=1.0.0&REQUEST=GetTile
  &LAYER=gis:my_layer
  &STYLE=&FORMAT=image/png
  &TILEMATRIXSET=EPSG:4326
  &TILEMATRIX=EPSG:4326:10
  &TILEROW=120&TILECOL=340

# Or use XYZ/TMS format (for Leaflet/MapLibre):
http://localhost:8080/geoserver/gwc/service/tms/1.0.0/gis:my_layer@EPSG:900913@png/{z}/{x}/{y}.png

Martin SQL tile functions

The most powerful Martin feature — write a PostgreSQL function that returns MVT tiles with custom logic, filtering, or joins.

sql — dynamic tile function
-- Create a tile function that accepts custom parameters
CREATE OR REPLACE FUNCTION gis.roads_by_type(
  z integer, x integer, y integer,
  query_params json   -- extra params from URL
) RETURNS bytea AS $$
DECLARE
  mvt bytea;
  road_filter text := COALESCE(query_params->>'type', 'primary');
BEGIN
  SELECT INTO mvt ST_AsMVT(tile, 'roads_by_type', 4096, 'geom')
  FROM (
    SELECT
      id, name, road_type, lanes,
      ST_AsMVTGeom(
        ST_Transform(geom, 3857),
        ST_TileEnvelope(z, x, y),
        4096, 64, true
      ) AS geom
    FROM gis.roads
    WHERE
      ST_Intersects(
        ST_Transform(geom, 3857),
        ST_TileEnvelope(z, x, y, margin => (64.0/4096))
      )
      AND road_type = road_filter    -- dynamic filter from URL param
  ) tile;

  RETURN mvt;
END;
$$ LANGUAGE plpgsql STABLE PARALLEL SAFE;

-- Call it from the browser:
-- http://localhost:3000/gis.roads_by_type/10/512/480?type=secondary
✅ SQL tile functions are automatically discovered by Martin if they match the (z int, x int, y int) signature. Add PARALLEL SAFE to allow PostGIS to parallelize tile generation across connections.

ogr2ogr Cheatsheet

bash — reference
## INSPECT
ogrinfo -al -so data.gpkg               # summary of all layers
ogrinfo data.gpkg layer_name            # detailed layer info

## FORMAT CONVERSIONS
# Shapefile → GeoPackage
ogr2ogr -f GPKG output.gpkg input.shp

# GeoJSON → PostGIS
ogr2ogr -f PostgreSQL "PG:..." data.geojson -nln table_name -lco SCHEMA=gis

# PostGIS → GeoPackage (export)
ogr2ogr -f GPKG output.gpkg "PG:host=localhost dbname=gisdb user=gisuser password=pw" \
  -sql "SELECT * FROM gis.my_layer WHERE category = 'urban'"

# PostGIS → GeoJSON
ogr2ogr -f GeoJSON output.geojson "PG:..." -sql "SELECT * FROM gis.my_layer"

## PROJECTIONS
ogr2ogr -t_srs EPSG:32737 reprojected.gpkg input.gpkg   # to UTM Zone 37S
ogr2ogr -t_srs EPSG:4326 wgs84.gpkg input.gpkg           # to WGS84

## CLIP / SPATIAL FILTER
ogr2ogr -clipsrc 36.5 -1.5 37.5 -0.5 clipped.gpkg input.gpkg   # bbox
ogr2ogr -clipsrc boundary.shp clipped.gpkg input.gpkg             # by polygon

## LARGE FILES — performance flags
ogr2ogr -f PostgreSQL "PG:..." huge.gpkg \
  --config PG_USE_COPY YES \     # COPY instead of INSERT (10x faster)
  --config OGR_TRUNCATE YES \    # Truncate before insert
  -gt 65536                       # Transaction batch size

Backup & Restore

bash — backup
# Full database backup (custom format)
make backup
# Creates: ./backups/gisdb_YYYYMMDD_HHMMSS.dump

# Schema-only backup
docker exec gis_postgis \
  pg_dump -U gisuser -d gisdb \
  --schema-only -Fc \
  > backups/schema_only.dump

# Single table backup
docker exec gis_postgis \
  pg_dump -U gisuser -d gisdb \
  -t gis.my_layer -Fc \
  > backups/my_layer.dump
bash — restore
# Full restore
make restore FILE=backups/gisdb_20241201.dump

# Restore single table
docker exec -i gis_postgis \
  pg_restore \
  -U gisuser -d gisdb \
  -t my_layer \
  --clean \
  < backups/my_layer.dump

# Cron job: daily backup at 2am
0 2 * * * cd /path/to/gis-stack && make backup

Debugging Common Issues

bash — diagnostic commands
## Container won't start?
docker compose ps                          # Check status
docker compose logs geoserver --tail 50    # Last 50 lines
docker inspect gis_geoserver | jq '.[0].State'  # Detailed state

## PostGIS not accepting connections?
docker exec gis_postgis pg_isready -U gisuser -d gisdb

## Martin shows no layers?
curl http://localhost:3000/catalog        # Check catalog
-- In psql: check geometry_columns
SELECT * FROM geometry_columns WHERE f_table_schema = 'gis';

## GeoServer layer error?
# Check Layer Preview → OpenLayers in GeoServer UI
# Check GeoServer logs:
docker exec gis_geoserver \
  tail -f /opt/geoserver/data_dir/logs/geoserver.log

## Slow spatial query?
-- In psql: check for missing index
EXPLAIN ANALYZE SELECT * FROM gis.my_layer
WHERE ST_Within(geom, ST_MakeEnvelope(36.7,-1.4,37.0,-1.1,4326));
-- If you see "Seq Scan" instead of "Index Scan" → create spatial index!

## Reset everything (nuclear option — DESTROYS DATA)
docker compose down -v   # -v removes volumes → data is gone!
make up
🔥 docker compose down -v will permanently delete your PostGIS and GeoServer data volumes. Always run make backup before using -v.
